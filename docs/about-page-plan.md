# About Me page ‚Äî implementation plan

A living "about me" page that pulls in real data from GitHub and Spotify, plus a CV
from a LinkedIn data export. (A health integration was evaluated and dropped ‚Äî see
Health below.) Tone: **playful-professional** ‚Äî credible enough to hire, fun enough
to remember.

## Decisions locked in

- **Fetching: hybrid.** Daily build/cron bakes most data to static JSON; Spotify
  "now playing" is the one live call.
- **CV source: LinkedIn data export.** Official archive ‚Üí parsed to `cv.json`. No
  scraping, no ToS issues.
- **Health: dropped.** Evaluated Fitbit, Google Health API, and Garmin ‚Äî all are
  either deprecating (legacy Fitbit, Sept 2026), gated behind Restricted-scope
  verification (Google Health API), or a paid partner program with OAuth 1.0a
  (Garmin Health API). None is a clean self-serve fit for a personal site. Whoop
  / Freestyle Libre remain possible future additions (self-serve OAuth 2.0; the
  user already has MCP servers for both), but the health section is shelved for now.
- **Tone: playful-professional.**
- **Host: Vercel Hobby (free).** Cron + functions both run on free. Verified:
  Hobby cron is daily-only with ¬±59 min precision ‚Äî fine for an about page. Functions
  get 300s max duration and full Node, so the live now-playing endpoint is fine.

## Host & routing (decided)

- **Backend host: GitHub Actions for the daily bakes + one Vercel function for
  now-playing.** Chosen for "robust + free": the rotating Fitbit refresh token needs
  writable persistent storage, and GitHub Actions can commit the rotated token + output
  JSON straight back to the repo (versioned, visible, zero extra infra) ‚Äî no Vercel KV
  to provision. No daily-frequency limit either. The single live piece, `/api/now-playing`,
  is a Vercel function (Spotify's refresh token doesn't rotate, so it's a clean env-var secret).
  - Open sub-detail: exact Fitbit-token persistence mechanism (encrypted committed file
    vs. Actions secret via API vs. free Upstash KV) ‚Äî pick during step 5; depends on
    whether the repo is public. Doesn't block earlier steps.
- **Routing: dedicated `/about` route** via react-router. Home links to it; footer gets
  an About link before Privacy.
- **Privacy folded into React.** The old standalone `public/privacy.html` is now a
  `/privacy` React route. All three pages (Home, About, Privacy) share one
  `src/components/Footer.tsx` (single source of truth). The animated gradient is scoped
  to the home page only (`.home` in app.css); footer + About + Privacy use static
  colours. Shared font (JetBrains Mono) site-wide; static pages use privacy's palette.

## Architecture

```
React SPA (Vite + TS, existing)
  /about reads baked JSON (cv, github, spotify-top)
  + ONE live fetch ‚Üí /api/now-playing

Daily bake (Vercel Cron OR GitHub Action)
  ‚Üí public/data/github.json
  ‚Üí public/data/spotify-top.json
  (secrets server-side only, never in client bundle)

cv.json ‚Äî generated by a local parser from LinkedIn export, committed manually
```

All secrets live in Vercel env vars / GitHub Actions secrets. Nothing sensitive
reaches the browser bundle.

## Data sources ‚Äî detail & gotchas

### CV (LinkedIn export ‚Üí cv.json) ‚Äî DONE (from Profile.csv)
- Source: LinkedIn ‚Üí Settings ‚Üí Data Privacy ‚Üí Get a copy of your data.
- `scripts/parse-cv.mjs <export-dir>` reads `Profile.csv` (and `Positions.csv` /
  `Skills.csv` if present) ‚Üí `public/data/cv.json`:
  `{ name, headline, location, summary, clients[], positions[], skills[] }`.
- Read at runtime via `src/data/cv.ts` (`useCVData()`); rendered in the
  "The serious bit" section (location, bio + personality tail as separate
  paragraphs, clients sentence, skill chips). Graceful null fallback.
- **Export gotcha:** LinkedIn strips line breaks from the Summary, gluing the
  "Core Technical Stack" block into unparseable text (e.g. `CSSBackend & CMS:`).
  The parser slices that block OUT (keeps clean intro + "When I am not‚Ä¶" tail)
  and uses a `CURATED_SKILLS` list instead ‚Äî robust, survives re-exports.
  `Skills.csv` overrides the curated list when present.
- **Pending enrichment:** drop `Positions.csv` + `Skills.csv` into the export
  dir and re-run to add the work-history timeline + real skill tags.
- Re-run manually whenever LinkedIn is updated. No runtime auth.

### GitHub (twice-daily bake) ‚Äî DONE
- `scripts/fetch-github.mjs`, dual-mode. With a PAT: GraphQL for pinned repos +
  language breakdown (count by primary language across all owned non-fork repos) +
  last-year contribution total. Tokenless fallback: public REST recent repos.
- **Finding:** no repos are currently pinned on the profile, so the script falls
  back to most-recently-updated repos when `pinnedItems` is empty (never shows an
  empty list). Pin some repos on GitHub to curate this.
- Output: `public/data/github.json`. Read at runtime via `src/data/github.ts`.
- Bake workflow: `.github/workflows/bake-github-data.yml` ‚Äî runs 06:00/18:00 UTC +
  manual, commits refreshed JSON to `main` only when it changes (`[skip ci]`).
- **Secret required:** `GH_DATA_TOKEN` (a PAT). NOT `GITHUB_TOKEN` ‚Äî that name is
  reserved by Actions and the built-in token can't read pinnedItems/contributions.

### Spotify (hybrid) ‚Äî DONE
- **Auth:** Authorization Code flow via `scripts/spotify-auth.mjs` (one-time local
  helper on 127.0.0.1:8888) ‚Üí long-lived refresh token captured.
- **Scopes:** `user-top-read`, `user-read-currently-playing`,
  `user-read-recently-played`.
- **Bake:** `scripts/fetch-spotify.mjs` ‚Üí top artists + tracks (`short_term`) ‚Üí
  `public/data/spotify-top.json`. Workflow `.github/workflows/bake-spotify-data.yml`
  runs 07:00/19:00 UTC + manual, commits to `main` on change.
- **Live:** `api/now-playing.js` Vercel function (refresh‚Üíaccess‚Üífetch), polled by
  `useNowPlaying` every 60s. Needs the three SPOTIFY_* env vars in Vercel.
- **Note:** `genres` removed ‚Äî Spotify now returns it empty on artist objects.
- **Secrets:** `SPOTIFY_CLIENT_ID/SECRET/REFRESH_TOKEN` as GitHub repo secrets
  (bake) AND Vercel env vars (live now-playing).

### Health ‚Äî DROPPED (evaluated Fitbit / Google Health / Garmin)
The "Keeping moving" section is shelved. Summary of why each source was rejected:
- **Legacy Fitbit Web API** ‚Äî easy self-serve OAuth, but **deprecated Sept 2026**.
  A full implementation was built and parked (draft PR #14, branch `feature/fitbit`)
  before being closed; refer to that branch's git history if revisiting.
- **Google Health API** (`health.googleapis.com`, the official Fitbit successor) ‚Äî
  the right data + server-side REST, but all scopes are **Restricted**, needing a
  Google Cloud project + OAuth consent verification before use. Not self-serve.
- **Garmin Health API** ‚Äî durable + ping/pull, but a **paid B2B partner program**
  ("contact us" application, commercial license fee) using **OAuth 1.0a**. The user
  wears Garmin but has no developer credentials, so the gating makes it a poor fit.

**Future options** (self-serve OAuth 2.0, user already has MCP servers for both):
Whoop (recovery/sleep/strain) or Freestyle Libre (glucose ‚Äî but it's medical data
on a public page, weigh privacy). If revisited, model the bake on the existing
GitHub/Spotify bakes and wire to a source-agnostic `health.json` / `useHealthData()`.

1. **Hero** ‚Äî name, one-line playful positioning, live "now playing" chip, quick links.

3. **What I build (GitHub)** ‚Äî pinned projects, language breakdown, contribution stat.

## Cross-cutting requirements

- **Graceful degradation:** every section renders fine if its JSON is missing/stale
  (API down, token expired). No section can break the page.
- **Styling:** match existing conventions in `src/app.css` ‚Äî CSS custom properties,
  grid/flex, `@keyframes`, `prefers-reduced-motion`. No new styling framework.
- **TypeScript:** typed data shapes for each JSON file; strict mode (matches tsconfig).
- **Privacy:** update `public/privacy.html` to note the page shows personal
  music/health stats.
- **Caching:** baked JSON served as static assets; now-playing endpoint sets short
  cache headers.

## Suggested build order

1. Routing/page scaffold + section layout with placeholder data.
2. CV parser + `cv.json` (no external auth ‚Äî quickest win).
3. GitHub integration (easiest API).
4. Spotify top + live now-playing.
5. Fitbit (incl. the rotating-refresh-token persistence design).
6. Polish: animation, responsive, dark mode, reduced-motion.
7. Secrets, cron wiring, deploy.
